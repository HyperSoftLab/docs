# MCP — Настройка

## Подключение

Включение MCP-эндпоинта на коллекторе.

<!-- STEP_GUIDE:START Включение MCP на коллекторе: переменные окружения, минимальная версия коллектора, проверка доступности эндпоинта. -->

Нужен коллектор GMONIT версии **>= v4-6274**.

Переменные окружения коллектора:

| Переменная | Описание | Пример |
|---|---|---|
| `MCP_ENABLED` | Включить MCP-модуль | `true` |
| `MCP_AUTH_TOKEN` | Bearer-токен для аутентификации MCP-клиентов. Любая непустая строка | `mcp-7Hq3...` |
| `CLICKHOUSE_READ_USER` | Пользователь ClickHouse, от имени которого MCP выполняет запросы | `gmonit_readonly` |
| `CLICKHOUSE_READ_KEY` | Пароль этого пользователя | `<секрет>` |

> При `MCP_ENABLED=true` `CLICKHOUSE_READ_USER` и `CLICKHOUSE_READ_KEY` становятся обязательными — без них коллектор не стартует. Кого выбирать в качестве этого пользователя — см. [Read-only пользователь ClickHouse](#read-only-пользователь-clickhouse).

После рестарта коллектора эндпоинт доступен по адресу `https://<адрес-коллектора>/mcp` (Streamable HTTP).

Быстрая проверка доступности:

```bash
curl -i -H "Authorization: Bearer <MCP_AUTH_TOKEN>" \
  https://<адрес-коллектора>/mcp
```

Коды ответа: `200`/`405` — эндпоинт жив (MCP принимает `POST`), `401` — неверный токен.

## Аутентификация по токену Grafana

Помимо общего токена MCP умеет пускать клиентов по персональному токену сервис-аккаунта Grafana — тогда каждый пользователь ходит под своей учёткой, а не под общим секретом. Оба способа работают одновременно.

<!-- STEP_GUIDE:START Аутентификация MCP-клиентов персональным токеном сервис-аккаунта Grafana как альтернатива общему токену: два принимаемых способа по логике ИЛИ, проверка токена обращением к Grafana, кеширование результата проверки и его время жизни, распространение смены прав и отзыва токена во времени, необходимые параметры подключения к Grafana. -->

MCP принимает Bearer-токен одним из двух способов (логика ИЛИ):

| Способ | Что за токен | Когда использовать |
|---|---|---|
| Общий токен | значение `MCP_AUTH_TOKEN` | сервисный/административный доступ, один секрет на всех |
| Персональный токен Grafana | токен сервис-аккаунта Grafana (`glsa_...`) | доступ под конкретным пользователем и с его правами |

Достаточно настроить **хотя бы один** способ. Если не задан ни `MCP_AUTH_TOKEN`, ни `GRAFANA_API_URL` — коллектор не стартует.

Переменные окружения для проверки токенов Grafana:

| Переменная | Описание | По умолчанию |
|---|---|---|
| `GRAFANA_API_URL` | Базовый URL Grafana (включая суффикс `/api`), к которой MCP обращается для проверки токена | — |
| `MCP_GRAFANA_AUTH_CACHE_TTL_MS` | Сколько миллисекунд кэшировать результат проверки токена | `60000` |

**Как это работает.** На входящий запрос MCP сперва сравнивает токен с общим `MCP_AUTH_TOKEN`; если не совпал — проверяет его как токен Grafana, обращаясь к Grafana API. Результат проверки (и успешный, и неуспешный) кэшируется на `MCP_GRAFANA_AUTH_CACHE_TTL_MS`, чтобы не обращаться к Grafana на каждый запрос.

**Как выпустить персональный токен.** В Grafana создайте сервис-аккаунт с нужной ролью (Viewer / Editor / Admin — влияет на доступные инструменты, см. [Разграничение прав](#разграничение-прав-роли)) и выпустите для него токен вида `glsa_...` в разделе сервис-аккаунтов администрирования Grafana. Этот токен клиент передаёт в заголовке `Authorization: Bearer glsa_...`.

**Смена прав и отзыв.** Изменение роли сервис-аккаунта или удаление токена в Grafana вступают в силу в пределах `MCP_GRAFANA_AUTH_CACHE_TTL_MS` (по умолчанию до минуты) — пока не истечёт кэш. После этого отозванный токен получает `401`.

## Разграничение прав (роли)

При аутентификации по токену Grafana набор доступных инструментов зависит от роли вызывающего в Grafana — низкопривилегированный пользователь не получит доступ к чувствительным операциям.

<!-- STEP_GUIDE:START Разграничение доступа к инструментам MCP по роли вызывающего (просмотр, редактирование, администрирование): какие группы инструментов доступны каждой роли, откуда берётся роль, ограничение произвольного SQL уровнем администратора, проксирование операций в Grafana под токеном самого пользователя и дополнительная проверка прав на стороне Grafana. -->

При аутентификации по токену Grafana MCP определяет роль вызывающего по его правам в Grafana и по ней ограничивает доступные инструменты. Уровни включают друг друга:

| Роль | Доступные инструменты |
|---|---|
| **Viewer** | Чтение: Apdex и здоровье приложений, проблемные приложения, состояние приложения, здоровье системы, схема таблиц; чтение из Grafana — аннотации, группы алертов, поиск и просмотр дашбордов |
| **Editor** | Всё из Viewer + изменяющие операции в Grafana: создание аннотаций, управление правилами алертов, изменение дашбордов |
| **Admin** | Всё из Editor + произвольный SQL (`execute_query`) |

- Роль определяется правами сервис-аккаунта в Grafana.
- Общий токен `MCP_AUTH_TOKEN` даёт полный доступ (эквивалент Admin).
- Изменяющие операции проксируются в Grafana **под токеном самого пользователя** — поэтому Grafana дополнительно проверяет его права (двойной контроль).
- Если роли не хватает прав на инструмент, агент получает сообщение вида `Forbidden: tool '<имя>' requires role '<роль>'` и сообщает вам.

## Read-only пользователь ClickHouse

MCP выполняет все запросы от имени пользователя ClickHouse, указанного в [подключении](#подключение). Чтобы AI-агент не мог изменить или удалить данные, выделите ему отдельную учётку с правом только на чтение.

<!-- STEP_GUIDE:START Использование выделенного read-only пользователя ClickHouse для MCP: мотивация, требуемые права на стороне ClickHouse, взаимодействие с режимом произвольного SQL. -->

Особенно важно при включённом [режиме произвольного SQL](#режим-произвольного-sql) — read-only пользователь там единственная реальная защита от разрушительных запросов агента.

## Режим произвольного SQL

По умолчанию агент работает только через структурированные инструменты (Apdex приложений, схема таблиц, состояние pipeline). Произвольный SQL включается отдельно — это компромисс между гибкостью и безопасностью.

<!-- STEP_GUIDE:START Включение режима произвольного SQL: переменная окружения, ограничения по объёму результата, риски, рекомендуемая связка с read-only пользователем, ограничение инструмента ролью администратора при аутентификации по токену Grafana. -->

| Переменная | Описание | По умолчанию |
|---|---|---|
| `MCP_RAW_SQL_ENABLED` | Разрешить инструмент `execute_query` с произвольным SQL | `false` |

Если флаг не выставлен, инструмент в наборе MCP остаётся, но любой его вызов агенту возвращает сообщение о том, что произвольный SQL отключён — у агента не получится обойти ограничение, переформулировав запрос.

При включении:

- Результат любого запроса режется на 200 строках
- Агент может выполнить всё, что позволят права пользователя ClickHouse — поэтому включайте только в связке с [read-only пользователем](#read-only-пользователь-clickhouse)
- Полезно для ad-hoc исследования данных, для production-окружений лучше держать выключенным
- При аутентификации по [токену Grafana](#аутентификация-по-токену-grafana) инструмент доступен только роли **Admin** (см. [Разграничение прав](#разграничение-прав-роли))

## Аудит событий

MCP умеет вести структурированный журнал событий (аутентификация, вызовы инструментов, выгрузки данных, ошибки) для передачи в SIEM. По умолчанию журнал выключен.

<!-- STEP_GUIDE:START Журналирование событий MCP для SIEM: включение аудита, вывод в выделенный поток или отдельный файл, состав события и его обязательные поля, отделение потока аудита от эксплуатационного вывода приложения, ответственность среды исполнения за доставку, ротацию и хранение журнала. -->

| Переменная | Описание | По умолчанию |
|---|---|---|
| `MCP_AUDIT_ENABLED` | Включить журналирование событий MCP | `false` |
| `MCP_AUDIT_FILE` | Путь к файлу журнала аудита. Если не задан — журнал пишется в stdout | — |

Что попадает в журнал: аутентификация (успех/отказ), вызовы инструментов, выгрузки данных (с текстом SQL-запроса), отказы авторизации по роли, ошибки обработки.

Каждое событие — одна строка JSON (UTF-8) с обязательными полями: `timestamp` (с точностью до мс и часовым поясом), `event_type`, `source_system`, `severity`. Формат совместим с KUMA.

**Выделенный поток.** Задавайте `MCP_AUDIT_FILE` — тогда журнал аудита пишется в отдельный файл, изолированный от эксплуатационного и диагностического вывода приложения. Это позволяет SIEM контролировать сам факт поступления событий (алерт на прекращение аудита) и не смешивать аудит с прочими логами.

**Доставка, ротация и хранение — на стороне среды исполнения.** Приложение только дозаписывает события в файл (append) и не управляет ротацией. Передачу в SIEM, ротацию и срок хранения настраивайте средствами окружения — например, `logrotate` в режиме `copytruncate` плюс syslog-агент (rsyslog и подобные). Append-режим корректно переживает `copytruncate`.

## Подключение клиента

Подключение AI-агента к развёрнутому MCP-эндпоинту.

<!-- STEP_GUIDE:START Подключение AI-агента к MCP-эндпоинту: пример команды для Claude Code, проверка подключения, подключение нескольких стендов одновременно, удаление. -->

Для Claude Code (CLI):

```bash
claude mcp add gmonit https://<адрес-коллектора>/mcp --header "Authorization: Bearer <MCP_AUTH_TOKEN>" --transport http  
```

`<MCP_AUTH_TOKEN>` и адрес коллектора возьмите у администратора стенда.

Проверить подключение:

```bash
claude mcp list
```

Несколько стендов можно держать одновременно — каждому своё имя:

```bash
claude mcp add gmonit-dev https://<dev-collector>/mcp --header "Authorization: Bearer <TOKEN_DEV>" --transport http
claude mcp add gmonit-prod https://<prod-collector>/mcp --header "Authorization: Bearer <TOKEN_PROD>" --transport http  
```

Удалить:

```bash
claude mcp remove gmonit
```

Другие MCP-клиенты (Codex, IDE-плагины) подключаются так же — указывается транспорт `HTTP`, URL вида `https://<адрес-коллектора>/mcp` и заголовок `Authorization: Bearer <MCP_AUTH_TOKEN>`.
